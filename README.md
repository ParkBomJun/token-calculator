# 토큰 계산기 (token-calculator)

LLM 프롬프트의 **토큰 수와 API 비용**을 계산하고, **어떤 모델이면 충분한지 실측으로 판정**하는 웹 도구.
서버 없이 전부 브라우저 안에서 동작하며, API 키는 각 벤더 공식 API로만 직행한다 (CSP로 강제).

**▶ 바로 사용: https://parkbomjun.github.io/token-calculator/**

| 기능 | 요약 |
|---|---|
| 토큰 계산 | 토크나이저 8종 로컬 실측 + 실측 배율 보정 (정확도 배지) |
| 비용 계산 | 요청당/캐시 적용 비용, Anthropic count_tokens 무료 정확 측정 |
| 모델 추천 | 작업 유형×오류 허용도 → 능력 축·언어 축 기반 최저 비용 모델 (규칙 기반, AI 호출 없음) |
| 프롬프트 변환 | 권장 프롬프팅 언어로 변환 (키 없는 모드 / Haiku 즉시 변환) |
| A/B 블라인드 실험대 | 모델 vs 모델, 프롬프트 원본 vs 수정본 — 익명 대결·투표·전적. "벤치마크 말고 내 작업으로 판정" |

---

## 1. 왜 만들었나

이 프로젝트는 RisuAI(오픈소스 LLM 채팅 프론트엔드)의 토큰 계산 코드(`tokenizer.ts`)를
검토하던 작업에서 출발했다. 검토 과정에서 두 가지를 확인했다:

1. RisuAI 내장 토크나이저 화면은 토큰 ID 배열을 그대로 뿌리는 **개발자용 도구**라서,
   "이 프롬프트가 몇 토큰이고 얼마가 나오는가"라는 실사용 질문에 답하지 못한다.
2. 계산 로직 자체에도 버그가 있었다(아래 §5 참조). 앱에 종속된 코드를 고치는 것과 별개로,
   **앱 없이 누구나 쓸 수 있는 독립 도구**가 필요하다고 판단해 이 저장소를 만들었다.

캐릭터 봇 제작(프롬프트/로어북/페르소나 작성)에서는 토큰 수가 곧 비용이자 컨텍스트 예산인데,
기존 온라인 계산기는 대부분 OpenAI tiktoken 기반이라 Claude·GLM·DeepSeek 사용자에게는
숫자가 맞지 않는다는 실제 불편이 동기였다.

## 1.5 모델 추천 기능 (v0.2)

"간단한 작업인데도 습관적으로 최상위 모델을 쓰는" 낭비를 줄이기 위한 기능.
계획서(텍스트/파일)를 넣고 작업 유형·오류 허용도·월 실행 횟수를 고르면:

1. 모델마다 **자기 토크나이저로 실측한 토큰 수 × 공식 가격**으로 요청당/월 비용을 비교하고
2. 필요 등급(경량/중급/최상위)을 충족하는 모델 중 최저 비용을 추천하며
3. **최상위 모델 습관 대비 월 절약액**을 병기한다.

판단은 규칙 기반(작업 유형 × 오류 허용도 → 필요 등급)이라 서버도 AI 호출도 없다.
모델 카탈로그는 `public/models.json` — 가격·등급·확인일·출처를 데이터로 관리하며,
git 커밋만으로 갱신된다 (이 프로젝트의 "DB"는 git이다).

## 1.6 프롬프트 언어 변환기 (v0.3)

추천 결과가 "이 모델은 영어 프롬프트를 더 잘 이해한다"고 조언할 때, 실제 변환까지
이어주는 기능 (`src/convert.js`). 두 모드:

- **방법 ① 키 없음** — 변환 지시문(의도·구조 보존, 코드·고유명사 유지, 결과물 언어
  유지 지시 자동 첨부)을 생성해 복사. 사용자가 평소 쓰는 AI 채팅에 붙여넣고 결과를
  가져오면 된다. 변환 비용 0.
- **방법 ② 본인 키** — Anthropic Haiku로 즉시 변환. CSP 화이트리스트가
  `api.anthropic.com`뿐이므로 즉시 변환은 Anthropic 키만 지원한다 (보안 원칙 §6 우선).

두 모드 모두 변환 결과를 **추천 모델의 토크나이저로 재실측**해 전/후 토큰·요청당·월
비용을 비교하고, 방법 ②는 변환 비용이 몇 회 실행이면 회수되는지 병기한다. 토큰이
오히려 늘어난 경우엔 "이득은 비용이 아니라 지시 이해도"라고 정직하게 표시한다.
"결과물 언어 유지" 지시는 대상 모델이 확실히 따르도록 **대상 언어로** 첨부된다
(예: 영어 변환 시 `Always respond in Korean.`).

## 1.7 A/B 블라인드 실험대 (v0.4)

"벤치마크는 오염(과최적화)될 수 있으니 최종 판단은 내 작업으로 A/B 실측"이라는 이
프로젝트의 원칙을 기능으로 만든 것 (`src/blind.js`). 두 모델에 같은 프롬프트를 보내
**무작위 순서로 익명 제시** → 투표 → 모델명·토큰·비용 공개 → 브라우저에 전적 누적.
3회 이상 대결에서 저가 모델이 밀리지 않으면 "그쪽으로 충분"이라고 알려준다.
추천 기능을 먼저 돌리면 "추천 모델 vs 습관적 최상위" 대진이 자동 세팅된다.

- **호출은 벤더 직행 우선, OpenRouter 폴백** (`src/vendors.js`, v0.4.1) — 🔑 중앙 키
  패널에 벤더 키를 넣으면 그 벤더 모델은 공식 API 직행(중개 마진 0), 없으면 OpenRouter
  키로 폴백한다. 처음엔 OpenRouter 단일 경유로 만들었으나 "크레딧이 너무 빨리 소모된다"는
  실사용 피드백으로 하루 만에 직행을 열었다. OpenRouter 슬러그는 `models.json`의
  `openrouter` 필드, 직행 표기가 다른 모델(Gemini의 점 표기)은 `directId` 필드
  (gemini-3-pro는 OpenRouter에 텍스트 모델이 없어 직행 전용 — 2026-07-12 확인).
- **벤더 CORS 실측 (2026-07-12)** — 프리플라이트(OPTIONS + Origin) 실측 결과
  Anthropic·OpenAI·DeepSeek·GLM(bigmodel.cn/z.ai 양쪽)·OpenRouter 모두 허용, Gemini도
  전용 헤더(`x-goog-api-key`)로는 허용. 이 실측이 직행 모드의 기술적 근거가 됐다.
  직행 모델명·파라미터 주의점: GPT-5 세대는 `max_tokens` 대신 `max_completion_tokens`,
  Gemini는 사고 토큰(`thoughtsTokenCount`)을 출력에 합산해 과금 기준과 맞춘다.
- **프롬프트 대결 모드** (v0.4.2) — 모델을 고정하고 원본 프롬프트 vs 수정본을 익명
  대결시킨다. "모델이 못한 건가, 프롬프트가 부실한 건가"의 변인 분리 실험. 첫 실측에서
  Fable 5가 GPT-5.6에 진 것이 계기 — 상담 멘트에 토큰을 쓴 응답 스타일이 패인이었는데,
  이게 모델 한계인지 지시문 부재인지 가리려면 같은 모델로 프롬프트만 바꿔 재대결해야
  한다. 절약 지시문 프리셋(코드/글쓰기/일반) 제공, 공개 시 수정본의 출력 토큰 절감률
  병기, 전적 3회+에서 수정본이 안 밀리면 "모델 바꾸기 전에 프롬프트부터"라고 알려준다.
- **블라인드 무결성** — 투표 전에는 모델명·토큰 수·비용을 일절 표시하지 않는다
  (비용 차이만으로 어느 쪽이 비싼 모델인지 알 수 있기 때문). 단, 응답이 스스로
  정체를 밝히는 것("저는 Claude입니다")까지는 막지 못한다는 한계를 UI에 명시했다.

## 2. 어떤 문제를 해결하나

- **모델마다 토크나이저가 달라서 토큰 수가 크게 갈린다.** 같은 한국어 한 문장이 실측으로
  GPT(o200k) 21토큰 ~ Claude(구세대) 39토큰까지 벌어진다(약 1.9배). OpenAI 기준 계산기로
  Claude 비용을 어림하면 체계적으로 틀린다. → 모델별 실제 어휘 파일로 로컬 계산.
- **토큰 수만으로는 판단이 안 된다.** 입력/예상 출력/캐시를 합친 **요청당 USD 비용**과
  프롬프트 캐시 적용 시 절감액(읽기 0.1×, 쓰기 1.25~2×)까지 계산한다.
- **Claude 3+ 는 토크나이저가 비공개라 로컬 계산이 근사치다.** Anthropic 공식
  `count_tokens` API(무료)를 연동해 실제 청구 기준 토큰 수와 로컬 추정의 편차(%)를
  바로 확인할 수 있게 했다.

## 3. 기술·구조를 왜 그렇게 선택했나

### 웹(정적 페이지) vs 설치형 프로그램

| | 정적 웹 (채택) | 설치형(CLI/Electron) |
|---|---|---|
| 설치 | 불필요, URL만 | 런타임/배포 파일 필요 |
| 공유·폰 사용 | 링크 하나 | 사실상 불가 |
| 서버 비용 | 0 (GitHub Pages) | — |

토큰화는 WASM으로 브라우저 안에서 전부 처리 가능하므로(서버가 필요한 연산이 없음)
설치형이 주는 이점이 없었다. 유일한 걱정이던 API 직접 호출(CORS)도 Anthropic이
`anthropic-dangerous-direct-browser-access` 헤더로 공식 허용해서 해소됐다.

### 세부 선택

- **Vite + 바닐라 JS (프레임워크 없음)** — 화면이 1개고 상태가 단순해서 React/Svelte는
  과투자. 의존성이 적을수록 유지보수와 코드 검토(§5)가 쉽다.
- **토크나이저 이원화** — OpenAI 계열은 순수 JS인 `gpt-tokenizer`(정확, WASM 불필요),
  나머지(HuggingFace tokenizer.json / SentencePiece 형식)는 `@mlc-ai/web-tokenizers`(WASM).
  모두 dynamic import로 코드 스플리팅해서 선택한 모델의 어휘만 내려받는다.
- **타입별 Promise 캐시 구조** (`src/tokenizers.js`) — 원본 RisuAI 코드는 "현재 로드된
  토크나이저 1개"를 전역 싱글턴으로 들고 있어서, 서로 다른 모델의 계산이 동시에 일어나면
  잘못된 토크나이저로 인코딩되는 레이스가 가능했다. 이 저장소는 처음부터
  `Map<type, Promise<Tokenizer>>`로 설계해 레이스와 중복 다운로드를 구조적으로 차단했다.
- **토큰 계산(tokenizers.js)과 비용 계산(pricing.js)의 분리 + graceful fallback** —
  가격이 등록되지 않은 모델은 `estimateCost()`가 null을 반환하고 UI는 토큰 수만 보여준다.
  **검증하지 않은 가격을 하드코딩해서 틀린 비용을 보여주는 것보다, 안 보여주는 게 낫다**는
  원칙. 대신 사용자가 UI에서 요금을 직접 입력하면 localStorage에 저장되어 계산에 쓰인다.
- **API 키를 서버로 보내지 않음** — count_tokens 호출은 브라우저 → Anthropic 직행이고
  키는 localStorage에만 남는다. 정적 호스팅이라 애초에 키를 받을 서버도 없다.

## 4. 구현 중 겪은 문제와 해결

| 문제 | 해결 |
|---|---|
| 모델 추천 기능: "계획서 난이도를 AI로 판단"하면 토큰을 아끼려고 토큰을 쓰는 모순 + 판단 주체의 API 키 문제 발생 | **규칙 기반으로 전환** — 작업 유형×오류 허용도 → 필요 등급 매핑(정립된 업계 지식), 등급 충족 모델 중 실측 비용 최저를 추천. AI 판단은 사용자 본인 키로 돌리는 선택 기능으로 보류 |
| 모델별 토크나이저가 달라 단일 토큰 수로 비용을 비교하면 최대 1.9배 왜곡 | 비교표 생성 시 **모델마다 자기 토크나이저로 실측** (같은 어휘를 쓰는 모델은 1회만 계산해 공유) |
| 타사 모델 가격이 애그리게이터마다 다르게 표기됨 | OpenAI·DeepSeek은 공식 페이지에서 직접 확인, 나머지는 출처·확인일을 `models.json`에 기록. 미확인 값은 넣지 않음 |
| 추천표에서 "토크나이저 로드 실패"와 "가격 미등록"이 같은 표시로 뭉개짐 (Node 통합 테스트에서 발견) | 판정 라벨을 실측 실패/가격 없음/등급 미달로 분리 |
| GLM 직접 측정이 결제 문제로 불가 + OpenRouter 경유 측정은 토큰 수를 자체 정규화해서 보고할 위험 | OpenRouter 단계에 **교차검증 모델**(직접 실측이 이미 있는 Claude)을 포함 — 경유 배율이 직접 배율과 6/6 항목 0.0% 차이로 일치 → 네이티브 패스스루 증명 후 GLM 값 채택 (×1.000, 정확 확정) |
| 구세대 Claude 어휘로는 신형 모델 토큰 수가 부정확 (한국어 ×0.96, 영어·코드 ×1.5로 언어별 편차 극심 — 실측으로 발견) | 델타 측정법(1배/2배 차분으로 메시지 오버헤드 상쇄) 캘리브레이션으로 언어별 보정계수를 산출해 UI에 내장. 언어 자동 감지 → 배율 적용 → 정확도 배지(정확/보정됨/근사) 표시 |
| 로컬 HTML 더블클릭(`file://`)으로는 토크나이저 파일 fetch가 브라우저 보안 정책에 막힘 | 배포 형태를 GitHub Pages(정적 호스팅)로 확정. `main` 푸시 → Actions 자동 빌드·배포 |
| `@mlc-ai/web-tokenizers`가 `"type": "module"`을 선언하고서 실제로는 UMD 번들이라, Node(ESM)에서 named import가 실패 → 배포 전 스모크 테스트가 막힘 | 번들 파일을 `.cjs`로 복사해 CJS 인터롭으로 로드하는 우회로 테스트 진행. 브라우저/Vite 경로는 Vite의 CJS 인터롭이 처리하므로 실사용에는 영향 없음을 확인 |
| 공개된 Claude 어휘(claude.json)는 구세대(Claude 2) 것이라 Claude 3+ 실청구와 편차 발생 — 특히 한국어에서 과대 추정 | 로컬 값을 "추정치"로 명시하고, 공식 count_tokens API(무료) 연동으로 정확값·편차를 병기. CORS는 공식 허용 헤더로 해결 |
| 타사(GLM/DeepSeek 등) 요금을 공식 확인 없이 넣으면 틀린 비용을 보여줄 위험 | 미등록 모델은 비용 표시 자체를 생략하고, "가격 직접 입력" 기능으로 사용자가 채우는 구조로 전환 |
| 어휘 파일이 총 ~43MB (GLM·Llama3 각 8MB대) — 첫 로드 부담 | 모델 선택 시에만 해당 파일을 지연 로드. 한 번 로드되면 세션 내 재사용(브라우저 캐시 + Promise 캐시) |
| 저장소 공개 범위(Public) 결정 | 무료 Pages 호스팅은 공개 저장소 전용 → 민감 정보가 코드에 없음을 확인하고 사용자 승인 후 공개로 생성 |

## 4.5 역할 분담 — 사람이 한 것 / AI가 한 것

이 프로젝트는 사람(기획·설계·검증) + AI(Claude, 구현·기술 검증)의 협업으로 만들어졌다.
"AI로 만들었다"를 숨기는 대신, 누가 어떤 결정을 했는지 기록한다.

### 사람이 한 것 (박범준)

- **기획과 방향**: 프로젝트 발안, "RisuAI 앱 통합 → 독립 실행 도구"로의 방향 전환 결정,
  기능 우선순위와 버전 로드맵 결정
- **핵심 설계 원칙 수립**:
  - "모델 카탈로그의 DB는 git이다" (별도 서버 없이 데이터를 커밋으로 관리)
  - "벤치마크 과최적화(벤치딸)를 경계한다" → 오염 저항 소스 우선 + 3단계 양자화 +
    **최종 판단은 A/B 실측** — 이 원칙이 나중에 블라인드 실험대가 됐다
  - API 키 보안 원칙 (본인 키만, 직행만, 저장은 옵트인)
- **설계 결함 발견**: 모델 추천이 1차원 티어라서 "Gemini Pro가 승자독식"하는 문제를
  지적 → 능력 축(reasoning/coding/writing/agent) 도입으로 이어짐
- **실측 데이터 생산**: 본인 API 키로 캘리브레이션 마법사를 돌려 72샘플 실측
  (Anthropic 4종·Gemini 2종·DeepSeek·GLM·GPT 교차검증)
- **실사용 검증과 버그 발견**: 블라인드 응답 칸 레이아웃 붕괴, Gemini 응답이 중간부터
  시작하는 문제(사고 요약 누출), 출력 한도 잘림, OpenRouter 크레딧 소모 문제
  → 전부 실사용 스크린샷과 함께 보고되어 수정됨 (§4 표 참조)
- **기능 발상**: 프롬프트 대결 모드("모델 탓인가 프롬프트 탓인가를 변인 분리"),
  응답 복사·미리보기·저장, 저장 파일 출처 기록 등

### AI가 한 것 (Claude)

- **코드 구현 전부**: Vite 앱 구조, 토크나이저 로더, 보정 엔진, 추천 엔진, 언어 변환기,
  블라인드 실험대, 벤더 직행 어댑터, 캘리브레이션 스크립트
- **기술 선택과 근거 제시**: 토크나이저 이원화, Promise 캐시 구조, CSP 설계,
  GPT-5 세대 `max_completion_tokens` 같은 벤더별 사양 대응
- **기술 검증**: 6개 벤더 CORS 프리플라이트 실측, OpenRouter 모델 슬러그 조회,
  참고한 원본(RisuAI) 코드의 버그 발견 (§5의 싱글턴 레이스·토크나이저 오매핑 등)
- **문서화**: 이 README 포함

### 검증 구조

AI가 짠 코드는 사람이 실사용으로 검증하고, 사람의 설계는 AI가 구현 전에 반론을 제기한다
(예: "전 벤더 CORS가 열려 있어도 CSP는 최소로" → 사용자가 비용 문제로 뒤집자 근거를
기록하고 개방). 어느 쪽도 상대 산출물을 무검증 통과시키지 않는 것이 규칙이다.

## 5. AI가 만든 코드를 어떻게 검토·수정하는가

이 저장소는 **사람(설계·방향 결정·최종 검토) ↔ AI(구현·사전 검증·설계 리뷰)** 의
교차 검토 방식으로 개발한다. 어느 쪽 산출물이든 상대가 검토한다.

**이 프로젝트의 출발점 자체가 AI 코드 검토였다.** RisuAI 원본 `tokenizer.ts`를 검토해
실제 버그를 찾아냈고, 그 교훈이 이 코드 설계에 반영됐다:

- llama3 선택 시 구형 llama 토크나이저로 계산되는 오매핑 (한 줄짜리 오타가 모든 llama3
  사용자의 토큰 수를 틀리게 만듦) → **이 저장소는 토크나이저 매핑을 switch 한 곳에 모으고
  스모크 테스트로 8종 전부의 결과를 실측**
- Gemma 토크나이저가 llama3 어휘를 로드하는 자산 불일치 → **어휘 파일 경로를 로더에
  명시적으로 1:1 대응시키고, 모델별 토큰 수가 서로 달라야 정상이라는 점을 테스트 관점으로 사용**
- 전역 싱글턴 레이스(§3) → **Promise 캐시 구조로 원천 차단**

이 저장소 코드에 대해 배포 전 실제로 수행한 검증:

1. **토크나이저 8종 실측 스모크 테스트** — 한국어+영어+이모지 혼합 문장을 8종 전부에
   통과시켜 결과가 (a) 오류 없이 나오고 (b) 모델별로 서로 다르며 (c) 알려진 경향(구형
   Claude가 한국어에서 최다 토큰)과 일치함을 확인. 실측값: o200k 21 / gemma 23 / llama3 24 /
   deepseek 27 / glm 29 / claude 39.
2. **빌드 검증** — `vite build` 성공 및 코드 스플리팅(어휘별 청크 분리) 확인.
3. **배포 후 검증** — 배포 URL의 페이지(200)와 어휘 파일 서빙(claude.json 200, 1.77MB)을
   직접 요청해서 확인.

한계도 기록해 둔다: 브라우저 UI 상호작용(버튼·캐시 토글 등)은 자동화 테스트가 아직 없고
수동 확인에 의존한다. vitest 기반 단위 테스트(비용 산식, 토크나이저 회귀) 추가가 다음 과제다.
AI가 작성한 코드에서 결함이 발견되면 이 README의 §4에 사례로 추가한다.

## 6. API 키 취급 원칙 (보안)

이 도구는 서버가 없으므로 API 키가 제작자에게 전송될 경로 자체가 없다. 구체적으로:

- **키는 항상 "사용자 본인의 키"** — 제작자 키를 사이트에 내장하는 방식은 공개 정적
  사이트에서 즉시 탈취·과금 악용되므로 채택하지 않는다.
- **통신은 사용자 브라우저 → 벤더 공식 API 직행** (Anthropic은 이 패턴을 공식 지원).
- **CSP 화이트리스트** — 페이지의 모든 네트워크 통신을 자체 호스트 + **벤더 공식 API
  도메인**(Anthropic·OpenAI·Google·DeepSeek·Z.ai/bigmodel·OpenRouter)으로 제한하는
  Content-Security-Policy를 선언했다. 만약 의존성 오염(공급망 공격)으로 악성 코드가
  끼어들어도 키를 내보낼 수 있는 목적지는 벤더 공식 API뿐이다. 새 벤더를 붙일 때마다
  이 목록에 명시적으로 추가해야 하며, 그 자체가 통신 대상의 감사 기록이 된다.
  (v0.4는 유출 표면 최소화를 위해 OpenRouter만 열었으나, 중개 크레딧 소모 문제로
  v0.4.1에서 사용자 결정으로 전 벤더 직행을 개방 — "임의 도메인 불가, 공식 API만"
  원칙은 유지된다. 각 키는 자기 벤더로만 전송된다.)
- **키 입력은 🔑 중앙 패널 하나** — 기능마다 흩어져 있던 키 입력을 한 곳으로 모아,
  어떤 키가 들어가 있고 어디로 가는지 한눈에 보이게 했다. 저장 옵트인도 단일 체크박스.
- **키 저장은 옵트인** — 기본값은 메모리(탭 닫으면 소멸), "이 브라우저에 키 저장" 체크 시에만
  localStorage. 공용 PC 사용을 고려한 설계.
- 권장: 이런 도구에 넣는 키는 지출 한도를 걸어둔 전용 키를 쓰는 것이 좋다.

## 실행

```bash
npm install
npm run dev        # 개발 서버 (http://localhost:5173)
npm run build      # dist/ 정적 빌드
npm run preview    # 빌드 결과 로컬 확인
```

배포: `main`에 푸시하면 GitHub Actions가 자동 빌드·배포 (`.github/workflows/deploy.yml`).

## 구조

```
index.html          UI
src/main.js         이벤트/렌더링
src/tokenizers.js   토크나이저 로더 (타입별 Promise 캐시) + count_tokens API
src/pricing.js      가격 테이블 + 비용 산식 (미등록 가격 → 토큰 수만 계산으로 폴백)
public/token/       토크나이저 어휘 파일 (~43MB)
```
